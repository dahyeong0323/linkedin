# LinkedIn Publisher Module Project Charter

## Purpose

Build a separate local LinkedIn Publisher Module that receives LinkedIn draft posts from the existing writing skill, stores them for review, lets the user edit or reject them, and publishes only explicitly approved text posts to the user's LinkedIn personal profile through the official LinkedIn API.

This module is not a replacement for the existing writing skill. The writing skill remains responsible for draft quality and style. The publisher module is responsible for draft storage, review workflow, approval gates, OAuth, publishing, and publish records.

## Current Project Context

- Repository/workspace root: current workspace root
- Existing writing skill location: `linkedin-style-writer-skill`
- New publisher module location: `linkedin-publisher`
- Existing writing skill modification policy: do not modify it unless a later phase explicitly requires and approves it.

## MVP Stack

- Language: Python
- Backend: FastAPI
- Storage: SQLite
- UI: minimal server-rendered web UI
- CLI: optional helper commands later
- Deployment target: local-only personal tool

## MVP Scope

Version 0.1 includes:

- Import or save draft text generated by the existing writing skill.
- Store drafts in SQLite with review status.
- List drafts.
- Show draft detail.
- Edit final text before approval.
- Request revision by recording a review state.
- Reject a draft.
- Approve a draft only through explicit user action.
- Publish only approved drafts as text-only LinkedIn personal profile posts.
- Store LinkedIn post URN, publish timestamp, final text, and publish errors.
- Write audit events for review and publishing actions.

## Explicit Non-Goals For v0.1

- No automatic publishing after draft generation.
- No scraping.
- No browser automation.
- No automated likes, comments, follows, connection requests, or engagement.
- No collection of other people's LinkedIn data.
- No company page posting.
- No image posting.
- No document or carousel posting.
- No scheduled posting.
- No multi-user SaaS behavior.
- No analytics or performance feedback loop.

## Publishing Rules

- Only `approved` drafts may be published.
- Approval must be an explicit user action.
- Draft generation must never imply approval.
- Publishing requires a final confirmation screen or equivalent explicit confirmation.
- A published draft must not be published again.
- If LinkedIn API behavior, permissions, scopes, or member URN handling are uncertain, implementation must stop and mark the item as `[NEEDS_VERIFICATION]` rather than inventing behavior.

## LinkedIn Target

MVP target is personal profile text-only posting.

Company page posting is excluded from v0.1.

## Analytics

Analytics are deferred to v0.2. Version 0.1 should store publish records but should not fetch or analyze post performance.

## Security Principles

- Use `.env` for local secrets.
- Never commit `.env`.
- Do not log LinkedIn client secrets, access tokens, or refresh tokens.
- Avoid plaintext token storage where feasible.
- Keep the app local-only by default.
- Do not publish source material directly. Publish only the reviewed final text.

## Phase Gate

Phase P0 is complete when this charter exists in `linkedin-publisher/PROJECT_CHARTER.md` and confirms:

- The publisher module is separate from the writing skill.
- The MVP is limited to local personal-profile text-only LinkedIn posting.
- Explicit approval is required before publishing.
- Scraping, browser automation, engagement automation, company page posting, and analytics are out of v0.1 scope.
